# `icon-[heroicons--*]` Spans Are Sized by `font-size`, Not `w-`/`h-` Utilities

> **Area:** Client | CSS | Tooling
> **Date:** 2026-05-29

## Context

The Home hero's "More info" button is icon-only on mobile (the label is `hidden sm:inline`).
The `information-circle` glyph looked far smaller than the shuffle glyph beside it, and bumping
its `w-5 h-5` тЖТ `w-6 h-6` changed nothing visible.

## What's going on

Icons are generated by `HurrahTv.Client/scripts/build-icons.mjs` into `wwwroot/css/icons.css`.
Each `.icon-[heroicons--name]` rule hardcodes:

```css
width: 1em;
height: 1em;
background-color: currentColor;
mask-image: url("data:image/svg+xml,...");
```

So **the glyph's size is `1em` тАФ i.e. the element's `font-size`** тАФ not the Tailwind `w-`/`h-`
utility on the span.

`index.html` loads `app.css` (line 28) *before* `icons.css` (line 29). `.w-6` and
`.icon-[тАж]` are both single-class selectors (equal specificity), so the later file wins тАФ
`icons.css`'s `width: 1em` overrides `.w-6`'s `width: 1.5rem`. **Every `w-N`/`h-N` on an icon
span across the app is a dead class.** They only *look* right because each button's
`font-size` happens to land near the intended px.

That's also why two adjacent icons can differ in size: the "More info" button sets `text-sm`
(тЖТ icon `14px` on mobile) while the shuffle button sets no font-size (тЖТ inherits the hero's
larger size). Same `w-5` on both, different rendered size.

## Resolution (#159)

**Fixed.** `build-icons.mjs` now emits the generated rules inside `@layer components`, so
Tailwind's `w-`/`h-` utilities (which v4 emits into `@layer utilities`) win тАФ `width: 1em`/
`height: 1em` are now *defaults* that apply only when no `w-`/`h-` utility is present.

The na├пve fix the original write-up suggested тАФ "emit the icon rule before Tailwind utilities"
(i.e. swap the `<link>` order) тАФ **would not have worked.** Tailwind v4 wraps utilities in
`@layer utilities`, and **unlayered styles outrank every layered style regardless of source
order.** So an unlayered `width: 1em` beats a layered `.w-5` even if `icons.css` loads first.
The cascade lever that actually works is *layer order*: app.css declares
`@layer theme, base, components, utilities;`, so putting the icon rules in `components` makes
the `utilities`-layer `w-`/`h-` win deterministically (and `hidden`/`flex` now override the
icon's `display: inline-block` too). `index.html` must keep `app.css` before `icons.css` so the
layer order is declared before the icon rules add to `components`.

## How to apply (post-#159)

- **Size an icon the normal Tailwind way: `w-N h-N`** (`w-5 h-5`, `w-6 h-6`). This is now the
  canonical, grep-able knob тАФ what every contributor already assumed worked.
- **`font-size` is still the fallback** for an icon span with no `w-`/`h-` class (`1em`). The
  HomeHero info + shuffle glyphs use `text-xl sm:text-lg` for responsive sizing and are
  unaffected тАФ but new icons should prefer `w-N h-N`.
- For an icon-only-at-one-breakpoint control, tie its horizontal padding to the *same*
  breakpoint that reveals the label (`px-3 sm:px-4` alongside `hidden sm:inline`) so the
  icon-only form stays a tight square.
